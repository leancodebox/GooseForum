# Resource V2 UI 设计规范

## 目的

本文档定义 GooseForum Resource V2 的可执行 UI 规范。

Discourse 是主要视觉和交互参考。现有 GooseForum 前台是历史实现，它借鉴了 Discourse，但没有形成完整设计系统。V2 应保留论坛产品的熟悉感、内容密度和交互质量，同时建立 GooseForum 自己可复用、可检查、可长期维护的规则。

实现时如果遇到本文档没有覆盖的 UI 决策，不应直接在页面代码里临时发挥。应该先补充或修正文档，再进入实现。

## 产品气质

GooseForum V2 应该给人的感觉：

- 信息密度高，但不拥挤。
- 安静、内容优先。
- 容易快速扫读。
- 适合长时间阅读。
- 适合每天重复使用。
- 对 Discourse 用户有熟悉感，但不是克隆。

应避免：

- 营销式 hero 区。
- 装饰性背景。
- 大量卡片堆砌的 landing page 结构。
- 运营工具区域使用过大的标题字号。
- 低信息密度的大面积留白。
- 每个页面各写一套临时视觉处理。

## 设计 Tokens

### 断点

默认使用 Tailwind 断点，除非项目配置明确调整：

```text
sm: 640px
md: 768px
lg: 1024px
xl: 1280px
2xl: 1536px
```

布局行为：

- `< lg`：移动端/平板使用抽屉导航。
- `lg`：展示左侧边栏。
- `xl`：文章详情可展示右侧信息栏。
- `2xl`：内容宽度不应无限拉伸。

### 布局宽度

```text
应用最大宽度: 1600px
左侧边栏: lg 210px, xl 224px
文章右侧栏: 280px
主内容横向 padding: 移动端 16px, sm 24px, lg 32px
```

规则：

- 主应用区域居中并限制最大宽度。
- 列表页使用“左侧边栏 + 主内容”。
- 详情页使用“左侧边栏 + 主内容 + 可选右侧栏”。
- 右侧栏只在能提供导航或上下文价值时出现。
- 当前已验收实现使用更紧凑的桌面左侧栏宽度，以优先保证 topic stream 和文章正文的可读宽度。

### 间距

使用克制的间距尺度：

```text
xs: 4px
sm: 8px
md: 12px
lg: 16px
xl: 24px
2xl: 32px
3xl: 48px
```

规则：

- 列表行垂直 padding：12px。
- 面板 padding：紧凑 16px，默认 20px，详情内容 24px。
- 侧边栏与主内容间距：8px。
- 主内容与右侧栏间距：24px。
- 避免页面分区出现过大的垂直空白。

### 圆角

```text
小控件: 6px
按钮/输入框: 8px
面板: 12px
头像: full
分类标识: 根据场景使用 3px 色条或 pill
```

规则：

- 重复内容面板可以使用 `rounded-xl`。
- 高密度列表行不要每一行都做成独立卡片。
- 不使用卡片套卡片。

### 边框

```text
默认边框: gray-100
强调边框: gray-200
交互边框: blue-500
危险边框: red-200
```

规则：

- 页面结构优先使用边框，而不是阴影。
- 列表行之间使用轻量分隔线。
- 表单 focus 状态结合 border 和 ring。

### 阴影

```text
面板阴影: 0 2px 12px rgba(0,0,0,0.04)
轻量卡片阴影: 0 2px 8px rgba(0,0,0,0.02)
浮层阴影: 0 4px 20px -4px rgba(0,0,0,0.1)
```

规则：

- 阴影必须克制。
- 不使用大面积装饰性阴影。
- hover 阴影只适合头像、浮层、菜单等局部对象。

### 字体

基础规则：

```text
正文: 14px 或 15px，视上下文决定
正文行高: 1.6
元信息: 12px 到 13px
主题标题: 移动端 15px，桌面端 16px
页面标题: 移动端 24px，桌面端 30px
分组标签: 11px，大写，粗体，适度 tracking
```

规则：

- 字号不随 viewport 宽度线性缩放。
- 默认 letter spacing 保持正常。
- 数字统计尽量使用 tabular numbers。
- 文章正文使用专门 prose 样式，不使用普通页面文本样式凑合。

### 颜色

中性色：

```text
页面背景: gray-50
表面: white
弱表面: gray-50/50
强文本: gray-900
默认文本: gray-700 或 gray-800
弱文本: gray-500
极弱文本: gray-400
边框: gray-100
```

品牌/操作色：

```text
primary: blue-600
primary hover: blue-700
primary subtle: blue-50
primary ring: blue-100 或 blue-500/20
```

状态色：

```text
success: green-500
warning: amber/yellow
danger: red-600
hot/accent: orange-400 或 orange-500
```

规则：

- 分类颜色只作为强调，不主导页面结构。
- 不使用大面积渐变和装饰性色块。
- active 导航要清晰，但不能刺眼。

### Z-Index

```text
sticky header: 50
移动端 drawer: 60
popover/menu: 50 到 70
全局通知: 100
加载条: 100
```

## App Shell

### 桌面结构

```text
body
  header sticky top
  main max-width 1600
    left sidebar
    page main
    optional right rail
```

规则：

- Header 高度 64px。
- Sidebar 在桌面端 sticky，顶部对齐 header 下方。
- 主内容不能被 sidebar 或 right rail 挤到不可读。
- 右侧栏只在 `xl` 及以上出现。

### 移动端结构

```text
body
  header sticky top
  main
  drawer navigation
```

规则：

- 移动端 header 更紧凑，但保留品牌、搜索入口、菜单入口。
- 左侧边栏进入 drawer。
- 右侧栏内容需要折叠到正文内或隐藏为次要信息。
- 移动端任何按钮文字都不能溢出容器。

## Header

Header 应包含：

- 品牌入口。
- 桌面端主导航。
- 搜索入口。
- 语言切换。
- 通知入口。
- 用户菜单。
- 登录/注册入口。
- 发布入口。
- 移动端菜单按钮。

行为：

- Header 保持 sticky。
- 站内 SPA 跳转时 header 不重建，只更新必要状态。
- 当前页面标题不应在 header 和页面标题之间重复造成噪音。
- 登录态与未登录态布局要稳定，不能造成明显宽度跳动。

## Sidebar

侧边栏由 layout payload 提供数据，前端统一渲染。

分区：

- 主导航。
- 资源入口。
- 分类列表。
- 页脚辅助链接。

状态：

- 当前页面 active。
- hover。
- focus。
- disabled 或不可访问。
- 分类 unread/new 可选状态。

规则：

- 每个页面不能重新实现自己的侧边栏。
- active 状态来自服务端提供的信息架构或前端路由匹配，不写死在模板里。
- 分类颜色只作为小色条或小圆点。

## 主题列表页

适用于：

- 首页。
- 分类页。
- 搜索结果页。
- 用户公开主题列表。

### 结构

```text
page
  optional announcement
  topic list panel
    toolbar
    topic rows
    pagination/load more
```

### Toolbar

常见入口：

- 最新。
- 热门。
- 精华或流行。
- 分类过滤。
- 新建主题。

规则：

- tab 使用清晰 active 状态。
- 新建主题是主要操作，但不能压过内容本身。
- 移动端工具栏可以横向滚动或折叠。

### Topic Row

必须支持的信息：

- 标题。
- 摘要或最后回复片段。
- 分类。
- 标签。
- 回复数。
- 浏览数。
- 最后活跃时间。
- 参与者头像。

桌面端建议列：

```text
main topic info | replies | views | activity
```

移动端：

- 标题占主导。
- 统计信息合并为一行 meta。
- 头像和最后活跃时间保留。
- 不强行保留桌面表格列。

视觉规则：

- 列表是高密度信息区域，不做每行大卡片。
- hover 可以改变背景色。
- unread/new/hot 可以使用小标识。
- 标题可换行，但最多两行，避免撑爆列表。
- 空状态要说明原因，并提供可执行入口。
- 加载状态不能造成布局大跳动。

## 文章详情页

### 结构

```text
article page
  title block
  category/meta
  post stream
  reply composer
  optional right rail
```

### 标题区

包含：

- 文章标题。
- 分类和标签。
- 作者。
- 创建时间/更新时间。
- 回复数、浏览数。
- 收藏、分享等操作。

规则：

- 标题是页面主视觉，但不要像营销页一样巨大。
- 分类 pill 和标签应克制。
- 移动端操作按钮可以折叠进菜单。

### Post Stream

每条回复结构：

```text
avatar | content column
         author/meta row
         body
         actions
```

规则：

- 桌面端头像 44px。
- 内容列最大宽度保证阅读舒适。
- 正文使用统一 prose 样式。
- 操作区默认低调，hover 或 focus 时增强。
- 移动端头像和作者信息仍要清晰。

### Reply Composer

登录用户：

- 展示回复入口。
- 支持 markdown。
- 支持预览。
- 支持提交中、失败、成功状态。

未登录用户：

- 展示登录/注册引导。
- 不显示不可用的假编辑器。

### Right Rail

可包含：

- 主题地图。
- 回复时间线。
- 频繁参与者。
- 推荐主题。
- 分享入口。

规则：

- 只在 `xl` 以上展示。
- 不承担核心内容。
- 移动端可折叠或隐藏。

## 分类页

分类页继承主题列表规范，并增加分类头：

必须支持：

- 分类名称。
- 分类颜色。
- 描述。
- 主题数量。
- 排序或筛选。

规则：

- 分类头不能做成大 hero。
- 分类颜色只作为识别辅助。
- 分类描述过长时需要折叠或限制宽度。

## 搜索页

必须支持：

- GET 搜索表单。
- 搜索关键词回显。
- 结果数量。
- 结果列表。
- 空状态。

规则：

- 搜索表单在 no-js 下也可用。
- 搜索结果使用主题列表行的变体。
- 高亮关键词要克制，不能破坏可读性。

## 用户主页

必须支持：

- 头像。
- 用户名。
- 简介。
- 注册时间或活跃信息。
- 统计数据。
- 主题/回复列表。
- 关注或私信入口，视权限决定。

规则：

- 公开信息在 no-js 下可读。
- 私有操作需要登录态校验。
- 用户页不要做成社交媒体主页，仍以论坛活动为主。

## 表单

所有表单必须支持：

- label。
- placeholder。
- help text。
- error text。
- focus。
- disabled。
- pending。
- success 或保存完成反馈。

规则：

- label 不应只依赖 placeholder。
- 错误信息放在对应字段附近。
- 提交按钮 pending 时防止重复提交。
- 禁用状态必须视觉明确。

## 发布与编辑

发布/编辑页属于后期高复杂页面，不应在基础架构未稳定时优先实现。

必须支持：

- 标题输入。
- 分类选择。
- Markdown 编辑器。
- 预览。
- 保存草稿。
- 发布/更新。
- 校验错误。

预览要求：

- 与文章详情页 prose 样式一致。
- 代码块、表格、引用、链接、图片样式一致。
- Mermaid 等增强内容可以在 JS 启用时渲染，no-js 下保留源内容或静态 fallback。

no-js：

- 至少提供基础发布表单，或明确说明需要启用 JavaScript。
- 不能出现空白页面。

## 交互状态

所有可交互元素都应定义：

- default。
- hover。
- active。
- focus-visible。
- disabled。
- loading。
- error。

站内跳转：

- 顶部可使用细加载条。
- 页面主体不应整体闪白。
- 旧内容可以短暂保留，直到新 payload 到达。
- 错误时回退到原生跳转或展示错误页。

数据变更：

- 可以使用乐观更新，但必须能回滚。
- 成功反馈使用轻量 toast 或局部状态。
- 失败反馈需要可读原因。

## 可访问性

要求：

- 页面使用语义化 heading。
- 链接和按钮语义正确。
- 菜单、drawer、modal 可键盘操作。
- focus 状态清晰。
- 图标按钮必须有 `aria-label` 或可访问文本。
- 图片有合适 alt。
- 颜色对比度满足基础可读性。
- 关键操作不能只依赖 hover 暴露。

## 国际化

规则：

- 通用 UI 文案由前端 i18n 或共享字典维护。
- 服务端 payload 只传业务数据，不每页重复传所有 label。
- 后端可以传业务实体名称、分类名、用户生成内容。
- 不在组件里硬编码英文。
- 中文、英文、日文长度差异都要纳入按钮和导航布局考虑。

## 页面验收清单

### 主题列表页

- 桌面端列布局清晰。
- 移动端信息不丢失。
- 标题、分类、统计、活跃时间完整。
- 空状态可理解。
- no-js 有核心列表。
- SPA 跳转无明显闪白。

### 文章详情页

- 标题区信息完整。
- 正文 prose 与预览一致。
- 回复流可读。
- 操作区状态清楚。
- 右侧栏不挤压正文。
- no-js 有标题、正文、回复核心内容。
- 禁用 JS 时页面不是空壳。

### 分类页

- 分类识别清晰。
- 主题列表与首页规则一致。
- 排序/筛选状态明确。
- no-js 可读。

### 搜索页

- GET 表单可用。
- 结果数量和关键词明确。
- 空状态明确。
- no-js 可用。

## 实施规则

实现过程中如果页面代码和本文档冲突，以本文档为准。

如果本文档不够具体，应先修改本文档，再写实现。这样 V2 才不会再次变成“看起来能跑，但每个页面都不太一样”的状态。
