# GooseForum 功能成本收益评估报告

本文档评估以下功能建议在 GooseForum 当前阶段的收益、成本、风险和建议优先级。

评估维度：

- 对论坛核心价值的提升
- 开发复杂度
- 后期维护成本
- 与当前产品阶段的契合度

## 总体结论

建议优先级如下：

```text
高优先级：
- 发帖被回复通知
- 文章发布/草稿状态切换

中优先级：
- 定时发布
- 后台数据导出
- 个人主页说说/笔记

低优先级或暂缓：
- 用户组可见标签
- 文章内容密码
- 任意修改文章日期
- 完整导入能力
```

推荐路线：

```text
第一批：
1. 文章作者回复通知
2. 草稿/发布状态切换

第二批：
1. 定时发布
2. 后台文章/用户导出

第三批：
1. 个人主页笔记/动态
2. 有约束的数据导入能力

暂缓：
1. tags 用户组权限
2. 文章密码
3. 任意文章日期修改
```

## 12. 用户组权限与部分标签可见

需求描述：

```text
增加用户组权限，例如部分 tags 只对某个用户组可见。
```

### 收益

收益中等偏低。

该能力适合以下场景：

- 私密社区
- 会员内容
- 内部知识库
- 分组讨论区
- 付费或邀请制内容

但对于公开论坛早期阶段，它不是最核心的互动能力。

### 成本

成本较高。

如果基于 tags 做可见性控制，会牵涉：

- 文章列表过滤
- 文章详情权限判断
- 搜索结果过滤
- 推荐列表过滤
- 缓存隔离
- SEO 和 sitemap
- 后台配置
- 前端不可见状态提示
- API 层防泄漏

其中最麻烦的是权限不能只做前端隐藏。所有列表、搜索、接口、缓存和 SEO 输出都必须遵守同一套规则，否则容易出现内容泄露。

### 风险

- 权限逻辑容易分散到多个查询中。
- tags 本身通常是内容标记，不一定适合作为权限边界。
- 搜索索引和缓存需要额外处理用户权限上下文。
- 后续维护成本高。

### 建议

暂缓。

如果未来要做权限内容，建议优先考虑“分类权限”，而不是 tags 权限。

原因是分类更接近信息架构和访问边界，tags 更适合作为内容标签。

## 13. 文章密码、日期修改、定时发布、发布与草稿切换

需求描述：

```text
增加文章内容密码、文章日期时间更改功能，包括选择未来时间/定时发布，以及发布与草稿状态切换。
```

该需求需要拆开评估，不能作为一个整体一起做。

## 13.1 发布与草稿状态切换

### 收益

收益高。

这是内容系统的基础能力：

- 作者可以保存草稿。
- 作者可以发布草稿。
- 作者可以下架自己的文章。
- 管理端可以更清晰地区分草稿和已发布内容。

当前模型中已有 `article_status`，具备实现基础。

### 成本

成本中低。

主要涉及：

- 发布页状态切换
- 编辑页草稿保存
- 列表查询过滤
- 后台文章状态展示和操作
- 权限校验

### 风险

风险可控。

需要注意：

- 草稿不能出现在公开列表。
- 草稿不能进入搜索索引。
- 作者自己可以查看和编辑草稿。
- 管理端是否可查看草稿需要明确权限。

### 建议

高优先级。

建议先做这个，不要和定时发布、文章密码混在一起。

## 13.2 定时发布

### 收益

收益中高。

适合公告、活动、教程更新、运营内容提前排期。

### 成本

成本中等。

需要新增类似字段：

```text
publish_at
```

公开查询需要满足：

```text
article_status = published
publish_at <= now
```

同时要处理：

- 首页列表
- 分类列表
- 文章详情
- 搜索索引
- sitemap
- RSS
- 缓存刷新

### 风险

- 时间和时区处理需要稳定。
- 定时发布前文章不能被公开访问。
- 搜索索引需要在发布时间后可见。
- 如果依赖缓存，发布时间到达后可能短时间不出现。

### 建议

中优先级。

可以作为发布/草稿切换后的第二阶段能力。

## 13.3 修改文章日期

### 收益

收益偏低。

该能力主要适合：

- 内容搬迁
- 归档整理
- 管理员修正历史文章时间

普通用户频繁修改文章日期的价值不高。

### 成本

成本中等。

文章日期可能影响：

- 排序
- RSS
- sitemap
- SEO
- 用户对“最新内容”的认知
- 管理审计

### 风险

如果允许普通用户随意修改时间，容易污染内容排序和时间线。

### 建议

暂缓。

如果要做，建议仅管理员可调整，并记录操作日志。

## 13.4 文章内容密码

### 收益

收益低到中等，取决于产品方向。

如果 GooseForum 要支持私密博客、半公开内容，文章密码有价值。

但如果产品核心是论坛讨论，文章密码不是优先能力。

### 成本

成本高。

需要处理：

- 列表摘要是否显示
- 搜索是否索引
- SEO 是否输出
- 缓存是否区分访问状态
- 文章详情解锁态
- 密码错误次数
- 分享体验
- 后台查看权限

### 风险

文章密码容易和公开论坛模型冲突，也容易和搜索、缓存、SEO 产生复杂边界。

### 建议

暂缓。

如果未来做私密内容，建议优先做更明确的权限模型，而不是单篇文章密码。

## 14. 个人主页说说 / 简单笔记

需求描述：

```text
增加个人主页的说说功能，类似简单笔记，可分为仅个人查看和公共查看，并显示在个人主页。
```

### 收益

收益中等。

它可以增强用户主页的活跃度，让用户除了发帖和回复之外，也能发布轻量内容。

适合：

- 个人状态
- 简短笔记
- 个人公告
- 公开动态

### 成本

成本中等。

需要新增：

- 数据模型
- 发布入口
- 编辑/删除
- 可见性控制
- 个人主页展示
- 管理端处理入口

如果支持评论、点赞、转发，复杂度会迅速上升。

### 风险

产品边界可能从论坛扩展到轻社交。

如果能力做得太多，会稀释论坛主线。

### 建议

可以做，但要控制范围。

第一阶段建议只支持：

```text
短文本或简单 Markdown
公开 / 仅自己可见
显示在个人主页
支持编辑和删除
```

第一阶段不建议支持：

```text
评论
点赞
转发
话题流
全站动态广场
```

建议命名为“个人笔记”或“个人动态”，避免产品预期过大。

## 15. 后台与前台数据导入导出

需求描述：

```text
增加后台到前端的数据导入和导出功能，并提供友好的操作。
```

### 收益

收益中高。

对站长很实用，尤其是：

- 内容备份
- 数据迁移
- 批量运营
- 审计分析
- 从其他系统迁入

### 成本

导出成本中等，导入成本高。

导出相对简单：

- 文章导出
- 用户导出
- 分类导出
- 回复导出

导入复杂很多：

- ID 映射
- 用户归属
- 分类映射
- 附件处理
- 重复数据
- 失败回滚
- 导入预览
- 权限校验
- 数据格式版本

### 风险

- 导入错误会污染数据库。
- 大文件导入需要进度、失败记录和重试机制。
- 前台开放导入导出容易产生权限和安全问题。

### 建议

先做后台导出，再做导入。

推荐顺序：

```text
1. 后台导出文章 CSV/JSON
2. 后台导出用户 CSV/JSON
3. 后台导出分类与站点配置
4. 后台导入文章
5. 全站迁移导入
```

不建议第一阶段做前台导入导出。

导入能力应该从后台开始，并严格限制管理员权限。

## 16. 发帖后有人回复时通知作者

需求描述：

```text
发帖时，当有人回复时通知我。
```

### 收益

收益非常高。

这是论坛互动闭环的核心能力：

- 作者知道自己的主题有人回应。
- 用户更容易回访。
- 讨论更容易持续。
- 站点活跃度提升明显。

相比很多后台增强，它更直接影响论坛生命力。

### 成本

完整回复通知系统成本中高，但可以切成很小的一期。

推荐第一阶段只做：

```text
文章作者回复通知
```

也就是：

```text
A 发文章
B 回复文章
A 收到通知
```

### 第一阶段边界

第一阶段只做：

```text
只通知文章作者
只通知有人回复了你的文章
自己回复自己的文章不通知
只生成站内通知
不做邮件
不做 WebSocket 实时推送
不做回复某层楼时通知该层楼作者
不做参与者订阅
不做 @ 用户
不做通知聚合
不做关闭单篇文章通知
```

### 最小逻辑

创建回复成功后：

```text
if article.user_id != current_user_id:
    create notification(
      user_id = article.user_id,
      actor_id = current_user_id,
      event_type = reply_article,
      article_id = article.id,
      reply_id = reply.id
    )
```

### 验收标准

```text
1. B 回复 A 的文章，A 收到通知
2. A 回复自己的文章，A 不收到通知
3. 未登录用户不能触发回复通知
4. 被封禁或不可见文章不触发通知
5. 删除回复后，第一阶段不撤回通知
```

### 风险

如果一开始就做完整系统，会牵扯：

- 楼中楼通知
- 参与者订阅
- 通知设置
- 邮件通知
- 实时推送
- 去重聚合
- 删除后的通知状态

这会让范围失控。

### 建议

高优先级，但必须控制范围。

任务命名建议为：

```text
文章作者回复通知
```

不要命名为：

```text
回复通知系统
```

前者是一个可控闭环，后者会变成完整通知系统工程。

## 最终建议

当前阶段最值得投入的是：

```text
1. 文章作者回复通知
2. 草稿/发布状态切换
3. 定时发布
4. 后台导出
```

其中“文章作者回复通知”虽然心理压力较大，但只要边界收窄，它是收益最高、最能提升论坛活跃度的功能。

应避免早期投入：

```text
tags 用户组权限
文章密码
任意文章日期修改
完整导入系统
复杂个人动态系统
```

这些功能不是不能做，而是不适合在当前阶段优先做。
